# AGENTS.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

Reference deployments for Cofide Connect — a workload identity platform built on SPIRE/SPIFFE. The repo has three main areas:

- **`workloads/`** — shell scripts that provision workload clusters (local Kind or AWS EKS), configure Connect via `cofidectl`, and validate with a ping-pong demo.
- **`control-plane/deployment/aws/infra/stack/`** — Terragrunt-managed Terraform stacks that provision the AWS infrastructure for the Connect control plane.
- **`control-plane/deployment/aws/infra/modules/`** — Reusable Terraform modules referenced by the Terragrunt units above.
- **`control-plane/deployment/aws/k8s/`** — Helm values and install scripts that deploy Kubernetes controllers and the SPIRE server onto the EKS cluster provisioned by the infra stacks.

## Documentation structure

The control plane documentation is layered:

- **`control-plane/deployment/aws/README.md`** — architecture overview and entry point
- **`infra/stack/README.md`** and **`k8s/README.md`** — deployment walkthroughs and adaptation guides for each layer
- **Group READMEs** (`infra/stack/spire-server/README.md`, `infra/stack/connect/README.md`, `k8s/controllers/README.md`, `k8s/spire-server/README.md`, `k8s/connect/README.md`) — detail for each logical group of units
- **Unit READMEs** (`base/dns/`, `base/database/rds-instance/`, `base/database/iam-admin/`, `base/eks-cluster/cluster/`, `base/eks-cluster/controllers/*/`) — unit-specific detail, required values, and operational notes
- **Module READMEs** (`infra/modules/*/README.md`) — auto-generated by terraform-docs; do not edit directly

When making changes that affect documentation:
- Module descriptions → edit the `/** ... */` block at the top of the module's `main.tf`
- Unit operational notes → edit the unit's `README.md`
- Deployment sequence changes → update the relevant layer README and the top-level `README.md`

## Linting and formatting

`lint.sh` runs all linting and formatting tools across the `control-plane/` and `workload/` directories:

- `terraform fmt` on all Terraform modules and stack units
- `terragrunt hcl fmt` on all Terragrunt stacks
- `terraform-docs` on each module, writing output to `README.md` in the module directory. Module-level description prose is written as a `/** ... */` block comment at the very top of `main.tf` — terraform-docs picks this up as the header automatically.
- `shellcheck` on all shell scripts under `control-plane/` and `workload/`

Stack roots are discovered via `root.hcl` and module directories are expected to be siblings of each stack root named `modules/`. Running the script requires `terraform`, `terragrunt`, `terraform-docs`, and `shellcheck` to be installed; it will report any missing tools before exiting.

```sh
./lint.sh
```

## Workloads

### Setup

```sh
cd workloads/
cp config.env.example config.env   # populate with Connect API details
./get-cofidectl.sh                  # download cofidectl + Connect plugin
./prerequisites.sh                  # verify tooling and log in
```

`config.env` must be populated before running any deployment script. It sets `CONNECT_URL`, `CONNECT_TRUST_DOMAIN`, `CONNECT_BUNDLE_HOST`, `AUTHORIZATION_DOMAIN`, `AUTHORIZATION_CLIENT_ID`, `NAMESPACE`, and `LOCAL`.

### Running deployments

All scripts are run from the `workloads/` directory (they use `$SCRIPT_DIR` to locate files):

| Script | Description |
|--------|-------------|
| `./single-trust-zone-cofidectl.sh` | Single trust zone on Kind with OSS SPIRE |
| `./single-trust-zone-cofidectl-tf.sh` | Single trust zone on Kind using terraform-provider-cofide |
| `./federated-cofidectl.sh` | Two federated trust zones on Kind with Cofide SPIRE |
| `./federated-cofidectl-tf.sh` | Two federated trust zones using terraform-provider-cofide |
| `./multi-mesh-cofidectl.sh` | Two federated trust zones with Istio on Kind |
| `./single-trust-zone-cofidectl-eks.sh` | Single trust zone on an existing EKS cluster |
| `./federated-helm-tf-eks.sh` | Two federated trust zones on EKS via Helm |

Each script generates a unique ID for all resources (trust zones, trust domains, cluster names) to avoid collisions with other runs against the same Connect instance.

### `cofide.yaml`

The `cofide.yaml` at the repo root is the local cofidectl config. It is deleted and regenerated at the start of each deployment script — do not rely on its contents between runs.

### Templates and generated files

Scripts use `envsubst` to render templates from `workloads/templates/` into `workloads/generated/`. The `generated/` directory is gitignored except for a `.gitkeep`.

### Terraform modules

Reusable Terraform modules live under `control-plane/deployment/aws/infra/modules/`:

```
modules/
  vpc/                          # VPC, subnets, IGW, NAT gateway
  jump/                         # SSM jump instance
  dns/                          # Route53 hosted zone (AWS-only; Cloudflare delegation stays in the unit)
  eks-cluster/                  # EKS cluster, node groups, IAM, KMS, add-ons
  external-dns/                 # IAM role + Pod Identity for ExternalDNS
  cert-manager/                 # IAM role + Pod Identity for cert-manager
  aws-load-balancer-controller/ # IAM role + Pod Identity for AWS Load Balancer Controller
  rds-instance/                 # RDS PostgreSQL instance, KMS keys, security group; accepts ingress_security_groups map
  rds-iam-admin/                # IAM admin PostgreSQL role (resources only; provider config stays in unit)
  spire/
    database/                   # SPIRE PostgreSQL database and role (resources only; provider config stays in unit)
    iam-role/                   # SPIRE server IAM role with KMS and RDS permissions
  connect/
    bundle-bucket/              # KMS-encrypted S3 bucket for trust bundles
    bundle-distribution/        # CloudFront distribution + ACM certificate + Route53 alias records
    database/                   # Connect API PostgreSQL database and role (resources only; provider config stays in unit)
    iam-role/                   # Connect API IAM role (SPIFFE JWT auth via SPIRE OIDC, RDS, S3)
```

Each Terragrunt unit references its module via the `source` attribute in `terragrunt.hcl`. Two patterns are used:

- **`source = "${get_repo_root()}//...modules/foo"`** — the unit is a pure wrapper; the module's files are the Terraform root for that unit.
- **`source = "."`** — the unit adds something on top of the module (e.g. a deployment-specific provider or extra resources). The unit's own `.tf` files are its Terraform root and call the module as a child.

When `source` points to a module, Terragrunt also copies any local `.tf` files from the unit directory into the working directory alongside the module files. This is used for:
- **`cloudflare.tf`** in the `dns` unit — Cloudflare NS delegation records and provider config.
- **`providers.tf`** in the `iam-admin` and `spire-server/database` units — PostgreSQL provider configuration that depends on Terraform variables and data sources.
- **`provider_variables.tf`** — variables used only by the provider config, kept in the unit rather than the module.
- **`providers_override.tf.json`** — merges additional `required_providers` entries into the module's `versions.tf` without replacing it (Terraform loads `*_override.tf.json` files last and merges their content).
- **`node_egress.tf`** and **`jump_egress.tf`** in the `rds-instance` unit — egress rules added to the EKS node and jump instance security groups to allow outbound connections to the RDS instance on port 5432. These are kept in the unit rather than the module so the `rds-instance` module stays generic and free of EKS/jump-specific concerns. The `rds-instance` module itself accepts an `ingress_security_groups` map (description → `{security_group_id, description}`) and creates one ingress rule per entry.

### Configuration

Before running any Terragrunt command, create `common.local.hcl` from the example in the `stack/` directory:

```sh
cp common.local.hcl.example common.local.hcl
```

This file must define `aws_account_id`, `aws_region`, `tf_state_bucket_region`, `tf_state_bucket_name`, and `tf_state_key_prefix`. It is gitignored.

Each unit can also have its own `common.local.hcl` to override the defaults defined in that unit's `terragrunt.hcl`. Each unit directory contains a `common.local.hcl.example` documenting the available overrides.

### Running Terragrunt

```sh
# From a specific stack unit directory, e.g. base/vpc/:
terragrunt init
terragrunt plan
terragrunt apply

# To deploy all units in dependency order from the stack/ directory:
terragrunt run-all apply
```

Remote state is stored in S3 with locking via the native S3 lock file backend (`use_lockfile = true`). The AWS provider is generated per-unit with `allowed_account_ids` set to prevent accidental cross-account operations.

### Optional dependencies

Inter-unit dependencies are optional where possible. Each unit reads outputs from its dependencies but falls back gracefully when a dependency has not been deployed — either via `try()` on the dependency output (returning null) or by allowing the user to supply values directly in the unit's `common.local.hcl`. This means units can be deployed independently against existing infrastructure without requiring the full stack.

### IAM authentication mode

Units that create IAM roles for Kubernetes controllers (`external-dns`, `cert-manager`, `aws-load-balancer-controller`, `spire-server`) support two IAM binding modes, configured via `iam_mode` in the unit's `common.local.hcl`:

- **`pod_identity`** (default) — uses EKS Pod Identity; requires the `eks-pod-identity-agent` add-on.
- **`irsa`** — uses IAM Roles for Service Accounts; requires `enable_irsa = true` in the cluster unit and `oidc_provider_arn` to be set.

### Database initialisation units

Three units manage the database layer and must be applied in order:

1. **`base/database/rds-instance/`** — provisions the RDS instance. Accessing it from a developer machine requires an SSM port-forwarding tunnel via the jump instance (see `base/database/rds-instance/README.md`).

2. **`base/database/iam-admin/`** — connects as the RDS master user (via Secrets Manager password over the SSM tunnel) and creates an `iam_admin` database role with `rds_superuser` and `rds_iam`. **This unit must be applied before any other unit that grants `rds_iam` to any user.** Granting `rds_iam` triggers a pg_hba.conf reload in RDS that switches all SSL connections to IAM token authentication — after this point the master password can no longer be used over SSL. Subsequent applies of `iam-admin` use the `aws rds generate-db-auth-token` path (set `db_actual_host` in the unit's `common.local.hcl`).

3. **`spire-server/database/`** and **`connect/database/`** — connect as `iam_admin` using IAM tokens and create the `spire` and `connect` databases and roles. The SSM tunnel must be open and the IAM identity running Terragrunt must have `rds-db:connect` permission for `iam_admin`.

The `cyrilgdn/postgresql` provider is used for `base/database/iam-admin`, `spire-server/database`, and `connect/database`. All three units support connecting to a non-stack database by supplying credentials directly in `common.local.hcl` — see each unit's `common.local.hcl.example` for the available options.

### RDS IAM authentication ARN format

The `rds-db:connect` IAM action uses a non-standard ARN format that is easy to get wrong. The correct format is:

```
arn:aws:rds-db:<region>:<account>:dbuser:<DbiResourceId>/<db-username>
```

Note the **colon** between `dbuser` and the `DbiResourceId`, and the **slash** between the `DbiResourceId` and the username. Using a slash instead of the colon (`dbuser/<DbiResourceId>/<db-username>`) produces a policy that the IAM simulator incorrectly evaluates as allowed — because the simulator compares the policy ARN against the resource ARN you supply, both using the same wrong format — but that RDS rejects at authentication time because it constructs the ARN internally using the colon format.

### VPC access via jump instance

The EKS API server endpoint is private by default (`cluster_access_mode = "ssm"`). The `jump/` unit provisions an SSM jump instance in a private subnet for accessing both the EKS API server (via kubectl port-forwarding) and the RDS instance (via PostgreSQL port-forwarding). No inbound rules are required on the jump instance — SSM connects outbound via the NAT gateway.

### Subnet types

The networking stack provides three subnet tiers:
- **public** — public IPs, routed via Internet Gateway
- **private** — no public IPs, outbound via a regional NAT gateway; used for EKS nodes and the jump instance
- **intra** — no public IPs, no internet route; used for RDS and other internal-only resources

### Configuration pattern

Each component directory follows the same pattern as the Terragrunt stacks:

- **`values.yaml`** — static defaults checked into git.
- **`values.local.yaml.example`** — documents available overrides; copy to `values.local.yaml` and fill in your values. This file is gitignored.
- **`values.override.yaml.example`** *(connect-api only)* — documents persistent manual overrides (e.g. `envoy.auth.audiences`, `connect.initialRBAC`); copy to `values.override.yaml`. Merged last by `install.sh`, so it takes precedence over both `values.yaml` and `values.local.yaml`. This file is gitignored.
- **`install.sh`** — runs `helm upgrade --install` merging `values.yaml` and `values.local.yaml` (if present).
- **`generate-local-values.sh`** *(optional)* — reads Terragrunt outputs from the infra stacks and writes `values.local.yaml` automatically. Requires the relevant infra units to have been applied.

If the infra stacks were used to provision the cluster, generate the local values file automatically:

```sh
./generate-local-values.sh
./install.sh
```

Otherwise, populate `values.local.yaml` manually:

```sh
cp values.local.yaml.example values.local.yaml
# Edit values.local.yaml with your cluster name, region, VPC ID, etc.
./install.sh
```

### IAM authentication mode

The IAM mode is set in the Terraform stacks (`iam_mode` in each controller's `common.local.hcl`), not here. When using the default `pod_identity` mode no `values.local.yaml` changes are needed — the Pod Identity Agent injects credentials automatically. When using `irsa`, uncomment the `serviceAccount.annotations` block in `values.local.yaml` and set the role ARN from the Terragrunt output:

```sh
cd infra/stack/base/eks-cluster/controllers/aws-load-balancer-controller
terragrunt output role_arn
```

### cert-manager ClusterIssuer

The ClusterIssuer is applied via kustomize after cert-manager is installed. It follows the same local override pattern: `cluster-issuer.local.yaml` (gitignored) is applied as a strategic merge patch on top of the base `cluster-issuer.yaml`.

```sh
cd cert-manager/cluster-issuer

# If the infra stacks were used, generate from Terragrunt outputs:
./generate-cluster-issuer-local.sh admin@example.com

# Otherwise, populate manually:
cp cluster-issuer.local.yaml.example cluster-issuer.local.yaml
# Edit cluster-issuer.local.yaml with your email, AWS region, and Route53 hosted zone ID.

./apply.sh
```

The ClusterIssuer uses Route53 DNS01 challenges. The cert-manager IAM role (created by `infra/stack/base/eks-cluster/controllers/cert-manager/`) has the necessary `route53:ChangeResourceRecordSets` permission scoped to the hosted zone created by `infra/stack/base/dns/`.

### SPIRE server

The SPIRE server is deployed as two separate Helm releases from `https://charts.cofide.dev`. cert-manager must already be installed before `spire` since it acts as the upstream authority. Install `spire-crds` before `spire`:

```sh
cd spire-server/spire-crds
./install.sh

cd spire-server/spire
# If the infra stacks were used, generate from Terragrunt outputs:
./generate-local-values.sh <trust-domain> <oidc-subdomain> <ca-country> <ca-organization> <ca-common-name>
./install.sh
```

Both charts use pre-release versions (e.g. `0.28.3-cofide.3`). Use `--devel` when searching for the latest available versions with `helm search repo cofide/ --devel`.

The `spire/generate-local-values.sh` script reads from five infra stack units (`base/eks-cluster/cluster`, `base/dns`, `base/database/rds-instance`, `spire-server/database`, `spire-server/iam-role`) and writes `values.local.yaml`. Five values cannot be derived from Terragrunt and are required arguments:

- `trust-domain` — SPIRE trust domain (e.g. `connect.example.cofide.dev`)
- `oidc-subdomain` — subdomain for the OIDC discovery provider (e.g. `oidc-discovery`)
- `ca-country` — country code for the upstream CA certificate subject (e.g. `GB`)
- `ca-organization` — organisation name for the upstream CA certificate subject
- `ca-common-name` — common name for the upstream CA certificate subject

The upstream authority is cert-manager, configured entirely in `values.yaml` via `upstreamAuthority.certManager`. The chart creates the CA and the necessary RBAC automatically (`ca.create: true`, `rbac.create: true`) — no local values or external PKI infrastructure are required.

The KMS `keyIdentifierValue` is set to `$POD_NAME`, injected at runtime via the Kubernetes Downward API. Each replica creates its own KMS key identified by its pod name, so no persistent storage is needed — the server runs as a `Deployment`.

#### IAM policy notes

The `rds-db:connect` IAM resource ARN uses a colon between `dbuser` and the DbiResourceId, and a slash between the DbiResourceId and the username:

```
arn:aws:rds-db:<region>:<account>:dbuser:<DbiResourceId>/<db-username>
```

Using a slash throughout (`dbuser/<id>/<user>`) causes the IAM simulator to report the action as allowed while RDS rejects the token at authentication time — the simulator matches the wrong format on both sides. See the [RDS IAM authentication ARN format](#rds-iam-authentication-arn-format) section under control plane infrastructure.

The KMS key operations (`kms:DescribeKey`, `kms:GetPublicKey`, `kms:ScheduleKeyDeletion`, `kms:Sign`) are intentionally absent from the IAM role policy. SPIRE's default key policy grants `kms:*` directly to the SPIRE server role on every key it creates; a direct principal grant in the key policy is sufficient without a corresponding IAM statement.

### Connect API and Connect UI

The Connect API (`cofide/cofide-connect`) and Connect UI (`cofide/cofide-connect-ui`) are deployed from `https://charts.cofide.dev` into the `connect` namespace. Both components use an envoy sidecar for TLS termination, so cert-manager Certificate resources must be applied before the Helm charts are installed.

Each component follows the same pattern: apply the certificate, then install the chart.

```sh
# Connect API
cd connect/connect-api/certificate
./generate-certificate-local.sh   # reads zone_name from base/dns
./apply.sh

cd connect/connect-api
./generate-local-values.sh <idp-issuer> <idp-jwks-uri> <ui-subdomain>
./install.sh

# Connect UI
cd connect/connect-ui/certificate
./generate-certificate-local.sh <ui-subdomain>
./apply.sh

cd connect/connect-ui
./generate-local-values.sh <ui-subdomain> <oauth-client-id> <oauth-issuer>
./install.sh
```

The `connect` namespace must be created before applying certificates or installing either chart, using `connect/namespace/apply.sh`.

#### Connect API hostname pattern

The Connect API chart uses an envoy sidecar that routes incoming TLS connections by SNI. `connect.urlBase` is the base domain (the Route53 zone name); envoy routes three SNI hostnames derived from it:

- `connect.<urlBase>` — Connect API endpoint
- `connect-agent.<urlBase>` — agent endpoint
- `xds.<urlBase>` — Aggregated Discovery Service endpoint

`generate-local-values.sh` reads `zone_name` from `base/dns` and uses it as `urlBase`. The ExternalDNS annotation on the service lists all three hostnames (comma-separated), and the cert-manager Certificate in `certificate/` covers all three DNS names. The TLS secret name `connect-tls` is referenced in `envoy.auth.tlsSecretName`.

#### Connect API generate-local-values.sh

Reads from seven infra stack units (`base/eks-cluster/cluster`, `base/dns`, `base/database/rds-instance`, `connect/database`, `connect/bundle-bucket`, `connect/bundle-distribution`, `connect/iam-role`) and writes `values.local.yaml`. The trust domain is read from the deployed `spire` Helm release in the `spire-mgmt` namespace, so that release must be installed before running this script. Three values cannot be derived automatically and are required arguments:

- `idp-issuer` — issuer URL of the identity provider (e.g. `https://auth.example.com`)
- `idp-jwks-uri` — JWKS URI of the identity provider
- `ui-subdomain` — subdomain the Connect UI is served on (e.g. `app`); used to set `connect.allowedOrigins` for browser CORS requests

The `connect.connectTrustBundleStoreURL` is set to `https://<distribution_domain_name>` from the `connect/bundle-distribution` unit (the CloudFront distribution serving the trust bundle S3 bucket).

The database connection uses the `connect.datastore.rds` backend with IAM token authentication at runtime via the Connect API IAM role. `oidc.iamRoleARN` and `oidc.awsRegion` are read from the `connect/iam-role` and `base/eks-cluster/cluster` units respectively; `oidc.audience` is hardcoded to `sts.amazonaws.com`.

#### Connect UI generate-local-values.sh

Reads `zone_name` from `base/dns` and requires three arguments:

- `ui-subdomain` — subdomain for the Connect UI (e.g. `app`)
- `oauth-client-id` — OAuth client ID for the identity provider
- `oauth-issuer` — issuer URL of the identity provider

The cert-manager Certificate in `certificate/` covers `<ui-subdomain>.<zone>` and writes the TLS secret `connect-ui-tls`, referenced in `envoy.tlsSecretName`.
